Personal Project · SaaS · Backend Infrastructure

Multi-tenant SaaS API

2025 GitHub ↗
Overview

Production-grade multi-tenant REST API built to demonstrate Clean Architecture, SOLID principles, and the ability to ship scalable SaaS backend infrastructure. The project covers the full surface of a real SaaS product: tenant provisioning, JWT authentication, role-based access, Stripe billing, and Swagger documentation — all in a single deployable stack.

Tech Stack
NestJS 10 TypeScript strict PostgreSQL 16 TypeORM 0.3 Stripe API JWT (access + refresh) Swagger / OpenAPI 3.0 Docker + docker-compose Jest class-validator Joi
Architecture

Multi-tenancy via schema isolation

Each tenant gets its own dedicated PostgreSQL schema (e.g., tenant_acme, tenant_globex). The TenantInterceptor resolves the tenant from the X-Tenant-ID header on every request, attaches it to the request context, and services use SET search_path to scope all queries to the correct schema automatically.

Key Features
Schema-per-tenant isolation
Each tenant gets a dedicated PostgreSQL schema provisioned on creation. The TenantDataSourceService manages connections and the TenantInterceptor scopes every request automatically.
JWT auth with token rotation
Access tokens (15m) and refresh tokens (7d) with separate secrets. The refresh endpoint issues a new access token while rotating the refresh token — no long-lived sessions.
Stripe billing integration
Full Stripe flow: create customer, assign subscription, handle invoice.paid and customer.subscription.deleted webhooks, expose billing portal URL. Runs in Stripe test mode.
Three-role RBAC system
SUPER_ADMIN, ADMIN, and MEMBER roles enforced via custom @Roles() decorator + RolesGuard. Each endpoint declares its minimum required role declaratively.
Env validation with Joi
All environment variables are validated at startup using a Joi schema. The app refuses to start if any required variable is missing or malformed — no silent config errors in production.
Swagger / OpenAPI 3.0
Full API documentation auto-generated at /api/docs. Every endpoint, DTO, and auth guard is documented with request/response schemas.
Global exception filter
All errors are normalized into a consistent shape: { statusCode, message, error, timestamp, path }. No raw NestJS stack traces leak to the client.
Docker + docker-compose
PostgreSQL service defined in docker-compose.yml. Full production build also containerized. One command to spin up the entire local stack.

RBAC
SUPER_ADMIN
Full access to all resources across all tenants. Can provision and delete tenant schemas, manage plans, and access billing for any tenant.
ADMIN
Manage users and billing within their own tenant. Cannot access other tenant data or global operations.
MEMBER
Read-only access within tenant. Can update their own profile. Cannot manage users or billing.
Design Decisions
→ Schema per tenant over row-level isolation — stronger data boundary guarantees, simpler query logic (no tenant_id filters everywhere), and easier to migrate or offboard a single tenant's data.
→ Interceptor-level tenant resolution — tenant context is attached once at the request boundary and available to all downstream services without passing it as a parameter through every function call.
→ Joi schema validation at startup — fail fast on missing config rather than letting the app run in a broken state and fail at runtime when a feature is first hit.
→ Separate access and refresh token secrets — a compromised refresh secret doesn't automatically compromise short-lived access tokens, and vice versa.
